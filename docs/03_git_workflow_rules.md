# 版本提交与远端同步规则

更新时间：2026-03-14

## 1. 基本原则
- 小步提交：每一轮“可验证改动”都应提交。
- 先验证后提交：提交前至少完成本轮定义的最小验证。
- 及时推送：提交完成后尽快 push 到远端，避免本地堆积。
- 可追溯：提交信息必须说明改了什么、为什么改、如何验证。

## 2. 提交触发条件
满足以下任一条件即应提交：
- 新增或完成一个明确子任务（如新增一个模块骨架）。
- 修复一个可复现问题并验证通过。
- 文档、流程、接口定义有阶段性稳定结果。

## 3. 单次提交最小清单
每次提交必须包含：
1. 变更文件列表（尽量小）。
2. 直接依赖关系（本改动依赖哪些模块/文档）。
3. 已完成状态说明（功能或文档达到何种程度）。
4. 复现方式（命令、输入、预期输出）。

## 4. 提交信息模板
建议格式：
`<type>: <scope> - <summary>`

类型建议：
- docs: 文档/规范
- feat: 新功能
- fix: 修复
- refactor: 重构
- test: 验证/测试
- chore: 杂项维护

示例：
- `docs: governance - add mainline/live-status/git workflow baseline`
- `feat: block-attn - add causal mask datapath in simulink model`
- `test: m1 - add block-level vector regression and report`

## 5. Push 规则
- 默认分支：main。
- 每次提交后，若无阻塞（网络/权限），立即 push。
- push 失败时，必须在 `docs/02_live_status.md` 记录原因与重试时间。

## 6. 禁止项
- 不要把与当前任务无关的大量文件混入一次提交。
- 不要在未验证的情况下提交“看起来能跑”的改动。
- 不要使用破坏性 git 操作覆盖未备份成果。

## 7. 工作流建议（每日）
1. 从 `docs/02_live_status.md` 读取今日目标。
2. 完成一轮最小改动。
3. 本地验证。
4. 更新实时状态。
5. commit + push。
6. 写明下一步。
